Understanding the WA-Relay Skill for OpenClaw
=============================================

In the rapidly evolving world of autonomous agents, efficient communication channels are the backbone of utility. OpenClaw, a framework designed for sophisticated agentic workflows, has introduced a powerful tool known as the **wa-relay** skill. If you have been wondering how to bridge the gap between third-party WhatsApp users and your primary AI agent, this guide will provide a comprehensive breakdown of what this skill does and why it is a game-changer for your automation stack.

What is WA-Relay?
-----------------

At its core, the **wa-relay** skill (v0.2.0) acts as a specialized communication bridge. It is designed to route third-party Direct Messages (DMs) from WhatsApp to your main OpenClaw agent. Instead of your main agent struggling to process every incoming raw message directly, the relay agent intercepts, processes, and facilitates a structured handover. This ensures that your primary AI agent is not overwhelmed by simple queries and receives only what is deemed relevant by your pre-configured routing logic.

### The Workflow Explained

The beauty of the wa-relay skill lies in its multi-step pipeline. When a third party sends a message to your WhatsApp endpoint, the sequence follows this logical progression:

1. **The Relay Agent Intercepts:** The relay agent receives the WhatsApp message. Crucially, it sends a `NO_REPLY` signal back to the sender, effectively putting the sender in a ‘pending’ state while the system processes the request.
2. **Inter-Session Communication:** The relay agent forwards the content to the main agent using the `sessions_send` method. This is the magic behind the curtain; it allows two separate agents within the OpenClaw ecosystem to ‘talk’ to each other securely.
3. **Intelligent Notification:** Once the main agent receives the forwarded data, it identifies the owner and sends a notification. This notification is not just the raw message—it includes a **suggested response** generated by the main agent’s internal logic.
4. **Owner Intervention:** The final authority rests with the owner. You receive a notification on WhatsApp containing both the original message and a draft reply. You can approve the suggestion, tweak it for tone, or discard it entirely in favor of a custom message.
5. **The Final Response:** Once you make your choice, the main agent sends the selected response back to the third party, completing the loop.

Bypassing the Relay: The Direct Number Allowlists
-------------------------------------------------

Not every contact needs to go through a relay. The developers of the wa-relay skill accounted for this by implementing a ‘Direct Number Allowlist.’ By adding specific phone numbers to this list, those contacts can bypass the relay logic entirely. This means their messages go straight to the main agent as if you were communicating with the bot directly, providing a high-priority ‘fast track’ for essential users or trusted administrators.

Technical Requirements and Setup
--------------------------------

To implement this, your environment must be running **OpenClaw v2026.2.14 or later**, with the WhatsApp channel already configured. The setup is handled through two primary scripts:

* `setup.sh`: This script handles the heavy lifting, including workspace creation and patching the main agent’s `SOUL.md` file to ensure it understands how to act when it receives relayed messages.
* `configure.sh`: This generates the routing configuration, allowing you to define which contacts are treated as standard third-party relay users and which are granted direct access.

A Note on Security and Transparency
-----------------------------------

Because the wa-relay skill performs deep integration, it requires elevated access. The developers have been incredibly transparent about three specific actions the setup script performs:

**1. Auth Credential Sharing:** OpenClaw agents typically operate in isolated auth environments. To allow the relay to function, the setup script copies your `auth-profiles.json` from the main agent to the relay agent. Without this, the relay would be unable to connect to model providers, resulting in silent failures.

**2. Session ID Regex Patching:** There is currently a known bug (issue #16211) regarding how OpenClaw validates session IDs. Specifically, it rejects colons and plus signs, which are essential for formatting WhatsApp numbers (e.g., `agent:wa-relay:whatsapp:+15551234567`). The script temporarily patches the `SAFE_SESSION_ID_RE` regex to allow these characters. Note that the script creates backups (`.bak` files) and prompts for confirmation, ensuring you have a fallback if you wish to revert changes once the official fix is merged.

**3. SOUL.md Modification:** Your main agent’s `SOUL.md` (its behavioral blueprint) is updated to include a ‘Relay de WhatsApp’ section. This allows the main agent to maintain context when dealing with relayed conversations, ensuring the personality and instructions of your agent remain consistent regardless of how the message arrives.

Why Use This?
-------------

Using the wa-relay skill is ideal for power users who want to gatekeep their AI agent’s interactions. By acting as a filter, the relay agent provides a layer of human-in-the-loop verification, preventing accidental or poor-quality AI responses from being sent to third parties. It turns your WhatsApp into a sophisticated command center, allowing you to manage multiple conversations with the intelligence of an AI assistant at your fingertips.

If you are serious about scaling your agentic capabilities, the wa-relay skill is an essential piece of the puzzle. Just remember to review the generated configurations and the modifications to your `SOUL.md` to ensure that your agent behaves exactly how you intend it to.
