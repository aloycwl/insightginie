<p>The Speckit Swarm skill is a native implementation of an agent‑orchestration system built on top of OpenClaw’s tooling. Inspired by the oh‑my‑opencode paradigm, it provides a structured way to break down complex AI‑assisted tasks into smaller, parallelizable chunks while ensuring each piece is completed fully. The skill lives in the <code>skills/speckit-swarm/</code> directory and consists of several TypeScript modules that work together to detect trigger keywords, select appropriate agent personas, plan task decomposition, and enforce continuation until the work is done.</p>
<p>At its core, Speckit Swarm relies on four main components. The Ultrawork Detector watches for the keywords &#8220;ulw&#8221; or &#8220;ultrawork&#8221; in a user’s prompt and, when found, flags the request for parallel execution. Agent Personas are a collection of specialized system prompts that define how each AI agent should behave—ranging from relentless orchestration to deep work, design debugging, research, or fast scouting. The Task Planner takes a high‑level goal and splits it into logical chunks that can be executed concurrently, while the Continuation Enforcer monitors each chunk to guarantee that no subtask is left incomplete before the overall job is considered finished.</p>
<p>The skill’s file layout mirrors its functional separation. The <code>src/</code> folder holds the implementation: <code>ultrawork.ts</code> contains the detection and triggering logic; the <code>personas/</code> subdirectory exports individual agent definitions such as <code>sisyphus.ts</code> (the main orchestrator), <code>hephaestus.ts</code> (the deep worker), <code>oracle.ts</code> (design/debug), <code>librarian.ts</code> (research/docs), and <code>explore.ts</code> (fast scout). The <code>planner.ts</code> file houses the task‑decomposition algorithm, and <code>index.ts</code> serves as the entry point that ties everything together for external consumption.</p>
<p>When a user includes &#8220;ulw&#8221; or &#8220;ultrawork&#8221; in their request, the skill switches into Ultrawork Mode. In this mode, the system first cleans the prefix, then uses the <code>planTask</code> function to break the cleaned instruction into multiple chunks. Each chunk receives a label, a specific task description, and optionally a model and thinking depth tailored to the chunk’s nature. The orchestrator then launches these chunks via <code>parallel_spawn</code>, waiting for all of them to finish before aggregating the results into a final answer.</p>
<p>Speckit Swarm defines five distinct personas, each tuned for a particular style of work. Sisyphus, the main orchestrator, runs on the <code>minimax-m2.5</code> model with high thinking depth, focusing on relentless execution, parallel coordination, and todo tracking. Hephaestus, the deep worker, shares the same model and thinking depth but emphasizes autonomous completion of a full scope without hand‑holding. Oracle handles architecture decisions, bug hunting, and code review, also using <code>minimax-m2.5</code> with high thinking. Librarian, responsible for documentation lookup, code exploration, and pattern finding, runs on the slightly lighter <code>minimax-m2.1</code> model with medium thinking. Finally, Explore acts as a fast scout, using <code>minimax-m2.5‑highspeed</code> with low thinking to perform rapid greps, file lookups, and quick analyses.</p>
<p>Direct usage of a persona is straightforward. Developers import the <code>PERSONAS</code> object and the <code>buildTaskPrompt</code> helper from the skill’s module. They select a persona—for example, <code>PERSONAS.hephaestus</code>—construct a task prompt with <code>buildTaskPrompt({ task, persona: 'hephaestus' })</code>, and then invoke <code>sessions_spawn</code> with the generated prompt, the persona’s model, and its thinking setting. This pattern allows fine‑grained control when a single, specialized agent is sufficient for the job at hand.</p>
<p>In Ultrawork Mode, the flow is automated. After confirming that the input contains the trigger keyword via <code>shouldUseUltrawork(task)</code>, the caller invokes <code>planTask(task)</code> to obtain a plan object whose <code>chunks</code> array contains the parallel work items. The orchestrator then calls <code>parallel_spawn</code> with those chunks and a wait policy of &#8220;all&#8221;, ensuring that the final response is assembled only after every chunk has reported success. This automation eliminates the need for manual persona selection when the user signals a desire for massive parallelism.</p>
<p>The <code>planTask</code> function itself is the heart of the decomposition logic. It examines the task description, estimates its complexity, and decides how to split it into meaningful pieces. For a request like &#8220;Create a new API endpoint&#8221;, the planner might produce chunks labeled &#8220;spec&#8221; (write the API specification), &#8220;setup&#8221; (create routing and controller scaffolding), &#8220;validation&#8221; (add input validation), and &#8220;tests&#8221; (write unit and integration tests). Each chunk receives a concise task string that a persona can act upon, making the parallel execution both efficient and context‑aware.</p>
<p>To support the automatic detection and preparation of ultrawork executions, the skill exports three helper functions. <code>containsUltrawork(task)</code> returns a boolean indicating whether the trigger keyword appears. <code>cleanUltraworkTask(task)</code> strips the &#8220;ulw/&#8221; or &#8220;ultrawork/&#8221; prefix, returning the core task description. Finally, <code>prepareUltrawork(task)</code> orchestrates the whole process: it returns an object with a <code>shouldExecute</code> flag, the array of <code>chunks</code> ready for <code>parallel_spawn</code>, and the cleaned task string for logging or reporting purposes. This encapsulation lets developers embed ultrawork handling directly into their agent loops with minimal boilerplate.</p>
<p>Before launching parallel agents, Speckit Swarm performs a lightweight concurrency safety check. It classifies the incoming task into one of several strategies: projects, CLIs, or APIs that involve creating many new files are marked as safe for parallel execution (<code>PARALLEL</code>). Tasks that would modify the same file repeatedly—such as bug fixes or edits to an existing module—are downgraded to <code>SEQUENTIAL</code> or even <code>SINGLE</code> to avoid race conditions. The decision flow first gauges overall complexity (is this a large undertaking?), then checks for file‑level conflicts (will multiple agents touch the same source file?), and finally selects the appropriate execution strategy.</p>
<p>An illustrative example demonstrates this safety net. Suppose the user says &#8220;ulw create a new CLI&#8221;. The detector sees the keyword, the planner judges the task as large and file‑conflict‑free, and returns a plan with multiple chunks—each responsible for a different aspect of the CLI (command parsing, subcommand implementation, help text generation, packaging). Because none of the chunks share the same file, the orchestrator safely dispatches them via <code>parallel_spawn</code>. Conversely, a request like &#8220;ulw fix the login bug in auth.ts&#8221; would be flagged as potentially conflicting; the planner would either serialize the chunks or collapse them into a single chunk, ensuring that only one agent edits the critical file at any moment.</p>
<p>In summary, the Speckit Swarm skill equips OpenClaw with a sophisticated, configurable orchestration layer that turns vague, high‑level AI prompts into concrete, parallelizable work packages. By combining keyword‑driven ultrawork activation, a roster of purpose‑built personas, intelligent task decomposition, and built‑in concurrency safeguards, the system enables developers to harness the full power of multiple AI agents without sacrificing correctness or encountering costly merge conflicts. Whether you need a single deep‑focused worker or a swarm of simultaneous specialists, Speckit Swarm provides the scaffolding to make it happen reliably and efficiently.</p>
