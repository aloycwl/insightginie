<h2>What Is the Frankenstein Skill?</h2>
<p>The Frankenstein skill is an innovative approach to AI skill development that combines the best features from multiple existing skills to create superior, comprehensive solutions. Named after the famous literary character who assembled body parts to create a new being, this skill searches across multiple repositories, analyzes security, and synthesizes the optimal combination of features.</p>
<h2>Why Use the Frankenstein Approach?</h3>
<p>This skill is particularly valuable when:</p>
<ul>
<li>Multiple skills exist for the same purpose but each has different strengths</li>
<li>You want the best-of-breed combination rather than settling for one skill’s limitations</li>
<li>You’re building a comprehensive skill from fragmented, specialized components</li>
</ul>
<h2>Core Functionality</h2>
<p>The Frankenstein skill performs several key functions:</p>
<ul>
<li>Comprehensive search across multiple skill repositories</li>
<li>Security scanning of candidate skills</li>
<li>Safe analysis of approved skills</li>
<li>Feature comparison and synthesis</li>
<li>Vetting and quality assurance</li>
</ul>
<h2>Search Sources</h2>
<p>The skill searches extensively across multiple platforms:</p>
<ol>
<li><strong>ClawHub</strong> &#8211; Primary repository using the clawhub CLI</li>
<li><strong>GitHub</strong> &#8211; Searches for AI skill repositories and instruction files</li>
<li><strong>skills.sh</strong> &#8211; Dedicated AI skills search platform</li>
<li><strong>skillsmp.com</strong> &#8211; Skills marketplace</li>
<li><strong>Other sources</strong> &#8211; Anthropic examples, OpenAI configurations, LangChain templates, and AutoGPT repos</li>
</ol>
<h2>Security First Approach</h2>
<p>Security is paramount in the Frankenstein process. Each candidate skill undergoes rigorous scanning using skill-auditor:</p>
<ul>
<li>Skills scoring 7+ are considered SAFE and proceed</li>
<li>Skills scoring below 7 are marked RISKY and skipped</li>
<li>Only security-scanned components are included in the final build</li>
</ul>
<h2>Safe Analysis Process</h2>
<p>Approved skills are analyzed in sandwrap read-only mode to extract:</p>
<ul>
<li>Core features and functionality</li>
<li>Methodology and problem-solving approaches</li>
<li>Reusable scripts and tools</li>
<li>Unique strengths and differentiators</li>
<li>Identified weaknesses or gaps</li>
</ul>

<h2>Synthesis Process</h2>
<p>The skill combines winning approaches for each feature:</p>
<ul>
<li>Feature 1 methodology from Skill A</li>
<li>Feature 2 implementation from Skill B</li>
<li>Feature 3 approach from Skill C</li>
</ul>
<h2>Building the Frankenstein Skill</h2>
<p>The skill uses skill-creator to assemble the final product:</p>
<ol>
<li>Combine winning features</li>
<li>Resolve conflicts between different approaches</li>
<li>Write unified documentation</li>
<li>Include scripts from winning skills</li>
<li>Document sources and attribution</li>
</ol>
<h2>Vetting Loop</h2>
<p>Critical quality assurance through multiple passes:</p>
<ol>
<li>Pass 1: Initial read and active breaking attempts</li>
<li>Pass 2: Further testing and issue identification</li>
<li>Pass 3+: Continue until no significant issues found</li>
</ol>
<p>Each pass documents:</p>
<ul>
<li>Issues found</li>
<li>Fixes applied</li>
<li>Stability assessment</li>
</ul>
<h2>Human Review</h2>
<p>Final step before deployment:</p>
<ul>
<li>Show what came from where</li>
<li>Highlight resolved conflicts</li>
<li>Present vetting summary</li>
<li>Request final approval</li>
</ul>
<h2>Technical Requirements</h2>
<p>The Frankenstein skill requires specific tools and models:</p>
<ul>
<li>clawhub CLI for searching and installing</li>
<li>skill-auditor for security scanning</li>
<li>sandwrap for safe analysis</li>
<li>skill-creator for building</li>
<li>Opus or best thinking model for deep reasoning</li>
</ul>
<h2>Model Requirements</h2>
<p>Deep reasoning is essential for successful Frankenstein skills:</p>
<ul>
<li>Compare multiple skill approaches effectively</li>
<li>Identify subtle methodology differences</li>
<li>Synthesize best parts creatively</li>
<li>Catch security and quality issues</li>
</ul>
<p>Only use smaller models if explicitly requested for cost reasons.</p>
<h2>Limitations</h2>
<p>Understand the boundaries:</p>
<ul>
<li>Only combines publicly available skills</li>
<li>Skips skills that fail security scans</li>
<li>Cannot resolve deep architectural conflicts</li>
<li>Human judgment needed for final synthesis</li>
<li>Quality depends on available skills</li>
</ul>
